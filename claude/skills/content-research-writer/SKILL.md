# Content Research Writer

リサーチ・引用追加・フック改善でコンテンツ執筆を支援。

## 概要

ライティングパートナーとして:
1. **協働アウトライン作成**: アイデアを構造化
2. **リサーチアシスト**: 関連情報の検索と引用追加
3. **フック改善**: 読者を引き込むオープニング強化
4. **セクション別フィードバック**: 各セクションを随時レビュー
5. **声の保持**: ライティングスタイルとトーンを維持
6. **引用管理**: 参照を適切にフォーマット
7. **反復改善**: 複数ドラフトを通じて改善

## 使用シーン

- ブログ記事・記事・ニュースレター執筆
- 教育コンテンツ・チュートリアル作成
- ソートリーダーシップ記事
- ケーススタディ
- 技術ドキュメント（ソース付き）
- 適切な引用付きライティング
- フック・導入部の改善
- セクション毎のフィードバック取得

## 使い方

### セットアップ

```bash
# 記事専用フォルダ作成
mkdir ~/writing/my-article-title
cd ~/writing/my-article-title

# ドラフトファイル作成
touch article-draft.md

# Claude Codeを開始
```

### 基本ワークフロー

#### 1. アウトラインから開始

```
"[トピック]についての記事のアウトラインを作成してください"
```

#### 2. リサーチと引用追加

```
"[特定トピック]についてリサーチし、アウトラインに引用を追加してください"
```

#### 3. フック改善

```
"この導入部があります。フックをもっと魅力的にしてください。"
```

#### 4. セクション別フィードバック

```
"「Why This Matters」セクションを書き終えました。レビューとフィードバックをお願いします。"
```

#### 5. 洗練と磨き上げ

```
"ドラフト全体をフロー・明瞭さ・一貫性についてレビューしてください。"
```

## 詳細手順

### Phase 1: プロジェクト理解

質問事項:
- トピックと主な主張は？
- ターゲットオーディエンスは？
- 希望する長さ/形式は？
- 目標は？（教育、説得、娯楽、説明）
- 既存のリサーチやソースは？
- ライティングスタイルは？（フォーマル、会話的、技術的）

### Phase 2: 協働アウトライン作成

```markdown
# Article Outline: [タイトル]

## Hook
- [オープニング行/ストーリー/統計]
- [読者がなぜ気にすべきか]

## Introduction
- 背景とコンテキスト
- 問題提起
- この記事が扱う内容

## Main Sections

### Section 1: [タイトル]
- キーポイントA
- キーポイントB
- 例/証拠
- [リサーチ必要: 特定トピック]

### Section 2: [タイトル]
- キーポイントC
- キーポイントD
- データ/引用必要

### Section 3: [タイトル]
- キーポイントE
- 反論
- 解決策

## Conclusion
- 主要ポイントのサマリー
- Call to action
- 最終的な考え

## Research To-Do
- [ ] [トピック]のデータを見つける
- [ ] [コンセプト]の例を取得
- [ ] [主張]のソース引用
```

**アウトラインの反復**:
- フィードバックに基づき調整
- 論理的フローを確保
- リサーチギャップを特定
- 深掘りセクションをマーク

### Phase 3: リサーチ実施

トピックについてリサーチ依頼があった場合:

```markdown
## Research: [トピック]

### Key Findings

1. **[見出し]**: [概要] [1]

2. **[見出し]**: [概要] [2]

3. **Expert Quote**: "[引用]" - [名前], [所属] [3]

### Citations
[1] 著者. (年). "タイトル". 出版物
[2] 著者. (年). "タイトル". 出版物
[3] 著者. (年). "タイトル". 出版物

アウトラインのSection 2に追加しました。
```

### Phase 4: フック改善

導入部を共有された場合、分析と強化:

```markdown
**Current Hook Analysis**:
- うまくいっている点: [ポジティブ要素]
- 改善できる点: [改善エリア]
- 感情的インパクト: [現在 vs. 潜在]

**Suggested Alternatives**:

Option 1: [大胆な主張]
> [例]
*なぜ効果的: [説明]*

Option 2: [個人的ストーリー]
> [例]
*なぜ効果的: [説明]*

Option 3: [驚きのデータ]
> [例]
*なぜ効果的: [説明]*

**フックの質問**:
- 好奇心を生み出すか？
- 価値を約束するか？
- 十分に具体的か？
- オーディエンスに合っているか？
```

### Phase 5: セクション別フィードバック

各セクション執筆後、レビュー:

```markdown
# Feedback: [セクション名]

## What Works Well ✓
- [強み1]
- [強み2]
- [強み3]

## Suggestions for Improvement

### Clarity
- [特定の問題] → [改善案]
- [複雑な文] → [よりシンプルな代替]

### Flow
- [遷移の問題] → [より良い接続]
- [段落の順序] → [推奨される並び替え]

### Evidence
- [サポートが必要な主張] → [引用や例を追加]
- [一般的な記述] → [より具体的に]

### Style
- [トーンの不一致] → [あなたの声により合わせる]
- [語彙選択] → [より強力な代替]

## Specific Line Edits

Original:
> [ドラフトからの正確な引用]

Suggested:
> [改善版]

Why: [説明]

## Questions to Consider
- [考えさせる質問1]
- [考えさせる質問2]

次のセクションへ進む準備完了！
```

### Phase 6: ライターの声を保持

重要な原則:
- **スタイルを学ぶ**: 既存の執筆サンプルを読む
- **提案、置き換えない**: 選択肢を提供、指示は避ける
- **トーンを合わせる**: フォーマル、カジュアル、技術的、フレンドリー
- **選択を尊重**: 彼らの版を好む場合、サポート
- **強化、上書きしない**: ライティングをより良く、異なるものにしない

定期的に確認:
- "これはあなたらしく聞こえますか？"
- "これは正しいトーンですか？"
- "もっと[フォーマル/カジュアル/技術的]にすべきですか？"

### Phase 7: 引用管理

ユーザーの好みに基づいて参照を処理:

**インライン引用**:
```markdown
Studies show 40% productivity improvement (McKinsey, 2024).
```

**番号付き参照**:
```markdown
Studies show 40% productivity improvement [1].

[1] McKinsey Global Institute. (2024)...
```

**脚注スタイル**:
```markdown
Studies show 40% productivity improvement^1

^1: McKinsey Global Institute. (2024)...
```

### Phase 8: 最終レビューと磨き

ドラフト完成時、包括的フィードバック:

```markdown
# Full Draft Review

## Overall Assessment

**Strengths**:
- [主な強み1]
- [主な強み2]
- [主な強み3]

**Impact**: [全体的な効果評価]

## Structure & Flow
- [組織についてのコメント]
- [遷移の質]
- [ペース評価]

## Content Quality
- [議論の強さ]
- [証拠の十分性]
- [例の効果]

## Technical Quality
- Grammar and mechanics: [評価]
- Consistency: [評価]
- Citations: [完全性チェック]

## Readability
- Clarity score: [評価]
- Sentence variety: [評価]
- Paragraph length: [評価]

## Final Polish Suggestions

1. **Introduction**: [具体的改善]
2. **Body**: [具体的改善]
3. **Conclusion**: [具体的改善]
4. **Title**: [必要に応じて選択肢]

## Pre-Publish Checklist
- [ ] すべての主張にソース
- [ ] 引用がフォーマット済み
- [ ] 例が明確
- [ ] 遷移がスムーズ
- [ ] Call to actionあり
- [ ] タイポ校正済み

公開準備完了！ 🚀
```

## ファイル構成

推奨構造:

```
~/writing/article-name/
├── outline.md          # アウトライン
├── research.md         # 全リサーチと引用
├── draft-v1.md         # 初稿
├── draft-v2.md         # 改訂稿
├── final.md            # 公開準備版
├── feedback.md         # 収集したフィードバック
└── sources/            # 参考資料
    ├── study1.pdf
    └── article2.md
```

## ベストプラクティス

### リサーチ
- 引用前にソースを検証
- 可能な限り最新データを使用
- 異なる視点をバランス
- オリジナルソースにリンク

### フィードバック
- 具体的に何を望むか明示: "これは技術的すぎますか？"
- 懸念を共有: "このセクションが冗長な気がします"
- 質問: "これは論理的にフローしていますか？"
- 代替案を依頼: "これを説明する別の方法は？"

### 声
- ライティング例を共有
- トーンの好みを指定
- 良いマッチを指摘: "それは私らしいです！"
- ミスマッチをフラグ: "私のスタイルにはフォーマル過ぎます"

## ライティングワークフロー

### ブログ記事
1. 一緒にアウトライン
2. キーポイントをリサーチ
3. 導入部を書く → フィードバック
4. 本文セクションを書く → 各フィードバック
5. 結論を書く → 最終レビュー
6. 磨きと編集

### ニュースレター
1. フックアイデアを議論
2. クイックアウトライン（短い形式）
3. 1セッションでドラフト
4. 明瞭さとリンクをレビュー
5. クイック磨き

### 技術チュートリアル
1. ステップをアウトライン
2. コード例を書く
3. 説明を追加
4. 指示をテスト
5. トラブルシューティングセクション追加
6. 正確性の最終レビュー

## プロのヒント

1. **VS Codeで作業**: Web版Claudeより長文向き
2. **一度に1セクション**: 段階的にフィードバック取得
3. **リサーチを別保存**: research.mdファイルを保持
4. **ドラフトをバージョン管理**: article-v1.md, article-v2.md等
5. **声に出して読む**: フィードバックで不自然な文を特定
6. **締切設定**: "今日中にドラフトを終わらせたい"
7. **休憩を取る**: 書く、フィードバック、一時停止、改訂
